# Contributing to Comb

Thanks for looking. This document covers how to get set up, what the project
cares about, and how to get a change merged.

## Getting set up

```bash
git clone https://github.com/jedbridges/comb.git
cd comb
make test        # about a second, no simulator
make run         # build, install, launch on a booted simulator
```

You need Xcode 26 and Swift 6.1 or later. `Comb.xcodeproj` is generated by
[XcodeGen](https://github.com/yonaskolb/XcodeGen) from `project.yml` and is not
checked in, so run `make project` after adding a file or changing build settings.

`make run --demo` is not a thing, but launching the app with the `--demo`
argument seeds an in-memory community so you can work on the UI without joining
anything real.

## Where to make a change

```
CombCore/     Protocol, crypto, wire format. Foundation only, no I/O.
CombStore/    Event log, projections, queries. GRDB.
CombNet/      Relay connection and protocol state machine.
Comb/         The iOS app. SwiftUI only.
```

CombNet never imports CombStore and vice versa. They meet at an `EventSink`
protocol, wired together by `CommunitySession` in the app. Keeping that boundary
is what lets the protocol state machine be tested against a mock socket.

If a change can live in a package rather than the app target, put it there: that
is the code that can be tested in a second without a simulator.

## What this project cares about

Read the design principles in the [README](README.md) first. They are
constraints, and a pull request that violates one will be asked to change even
if it works.

Beyond those:

**Comments explain why, not what.** The code says what it does. A comment earns
its place by recording the reason a thing is the way it is, especially when the
obvious implementation was tried and failed. If you fixed something subtle,
write down what broke.

**No new dependencies without a conversation.** The app has two: swift-secp256k1
and GRDB. Both are load-bearing and neither has network surface. Open an issue
before adding a third.

**Tests for anything protocol-shaped.** Event ids, serialization, filters,
auth, storage predicates, and the relay state machine all have tests and should
keep having them. UI changes are verified by running the app; that is fine and
expected.

**Honesty in user-facing text.** If a feature has a limitation, the UI says so.
Comb tells people that notifications can be late and that blocking is local,
because both are true. Do not write copy that implies a guarantee the code
cannot keep.

## Making a pull request

1. Branch from `main`.
2. Make the change. Run `make test`.
3. If you touched the app, run it and look at the thing you changed.
4. Write a commit message that explains the reasoning, not just the diff. Recent
   history is the house style.
5. Open the pull request against `main`, describing what changed and why, and how
   you verified it.

CI runs the three package test suites plus an app build. It has to be green.

## Reporting bugs

Use the [bug report template](https://github.com/jedbridges/comb/issues/new?template=bug-report.yml).
From inside the app, **Settings → Report a problem** assembles a report with the
device, the version, and a local log; that log makes a bug far easier to trace.

For anything security-sensitive, do not open an issue. See
[SECURITY.md](SECURITY.md).

## Listing a community

You do not need to write code to get a community into Comb's browse screen.
[Open a listing request](https://github.com/jedbridges/comb/issues/new?template=list-community.yml)
with the name and relay address, and a maintainer will add it to
`communities/index.json`.

Note that the index is not signed, and cannot meaningfully be: signing it would
require baking a trust anchor into the app, which contradicts the no-privileged-relay
principle. TLS plus a publicly auditable git repository is the honest story.
Entries are never auto-joined.
